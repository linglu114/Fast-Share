## 一、总体架构

采用 **分层模块化** 架构，Flutter 跨平台 UI，核心逻辑用纯 Dart 实现。**发送和接收各有独立的 Engine Isolate**，与 UI 线程完全隔离，平台差异通过薄通道层抽象。

```
┌──────────────────────────────────────────────────────┐
│  UI Isolate (Flutter Main)                           │
│  ┌──────────────────────────────────────────────────┐│
│  │ Presentation (Widgets)                          ││
│  │ 设备与发现 | 传输 | 历史 | 设置                  ││
│  ├──────────────────────────────────────────────────┤│
│  │ State Management (Riverpod, 9 providers)        ││
│  └──────────────────────────────────────────────────┘│
├──────────────────────────────────────────────────────┤
│  Engine Isolates (Dart Isolate × 2)                  │
│  ┌────────────────────┐ ┌─────────────────────┐     │
│  │ TransferEngine     │ │ ReceiveEngine       │     │
│  │ (发送端)           │ │ (接收端)            │     │
│  │ FileScan / Read    │ │ FrameParse / Write  │     │
│  │ CRC32 / Chunk      │ │ ACK / DirCreate     │     │
│  │ TokenBucket        │ │ Resume              │     │
│  └────────────────────┘ └─────────────────────┘     │
├──────────────────────────────────────────────────────┤
│  Business Logic (Pure Dart, runs in UI Isolate)      │
│  ┌──────────┐ ┌───────────┐ ┌──────────────────┐    │
│  │ Discovery│ │ Connection│ │ Clipboard/History│    │
│  │ UDP 广播  │ │ Manager   │ │ /Settings        │    │
│  └──────────┘ └───────────┘ └──────────────────┘    │
├──────────────────────────────────────────────────────┤
│  Network Transport (Dart:io + custom FLP protocol)   │
│  UDP Broadcast Discovery | TCP Server/Client        │
├──────────────────────────────────────────────────────┤
│  Platform Abstraction Layer (MethodChannel)           │
│  Network | WiFi lock | File access | Clipboard       │
│  Permissions | Foreground Service                    │
├──────────────────────────────────────────────────────┤
│  Host Platform (Android 8+, Windows 10 1809+)        │
└──────────────────────────────────────────────────────┘
```

**关键设计原则**：

- UI 与逻辑完全解耦，业务模块通过 Riverpod 与 UI 通信
- **发送和接收分别运行在独立的 Engine Isolate 中**：`TransferEngine` 负责文件扫描、读取、分片、Socket 写入；`ReceiveEngine` 负责帧解析、目录创建、磁盘写入、ACK 生成
- UI Isolate 仅接收进度事件，不直接参与文件 I/O 或协议解析
- 平台通道仅暴露最小必要功能，业务不依赖平台

## 二、模块职责

### 2.1 设备发现模块 (DiscoveryService)

- 基于 **UDP 广播**：向 `255.255.255.255:45679` 及子网定向广播地址周期性发送 `fastshare_hello` JSON 数据报
- 维护在线设备列表（去重、心跳 15s 超时、离线移除）
- RFC 1918 子网广播地址计算（/8 /12 /16，而非硬编码 /24）
- Android 端通过 WiFi 多播锁保持后台可发现
- 输出设备上线/下线事件流

### 2.2 连接与配对管理 (ConnectionManager)

- 信任设备存储（SQLite）
- TCP 长连接池：发现连接 + 引擎连接双通道
- 配对流程：6 位随机码挑战，3 次握手，确认后持久化 SHA256 Token
- 接收端 Offer 管理：`acceptPendingOffer()` / `rejectPendingOffer()`
- 剪贴板中继：通过现有 TCP 连接推送文本

### 2.3 文件传输引擎

**发送端 — TransferEngine**（独立 Dart Isolate）：

- **文件扫描**：Engine 内部遍历目录，通过 `FileStat.size` 读取实际文件大小；支持 content URI 文件列表
- **传输模式判定器**：大文件（≥100MB）→ 顺序；全部小文件 → 并发；混合 → 先大后小
- **动态并发调整器**：初始并发 3~4，范围 3~8，每 5 秒根据吞吐量趋势调整 ±1
- **块传输控制器**：8MB 分块，CRC32 零校验和标记，滑动窗口 ACK（3 chunk = 24MB 在途）
- **文件夹处理**：以文件夹名作为相对路径前缀，完整保留目录结构
- **速率控制**：令牌桶算法，支持手动限速和动态调整
- **进度汇报**：向 UI Isolate 发送进度/速度/文件完成/模式变更事件
- **资源释放**：传输完成或队列为空后 Engine Isolate 自动销毁

**接收端 — ReceiveEngine**（独立 Dart Isolate）：

- 从 TCP Socket 流式读取 FLP Frame
- 控制帧 → 中继到 UI Isolate
- 数据帧 → 校验 payload 完整性，创建目录结构，写入磁盘，回复 ACK
- 文件完成 → 发送 FILE_COMPLETE；全部完成 → TRANSFER_COMPLETE
- 截断帧保护：payload 长度校验防止 RangeError

### 2.4 剪贴板共享模块

- 通过 TCP 连接推送文本，接收端通过 `Clipboard.setData()` 写入系统剪贴板

### 2.5 历史记录与设置

- **历史记录**：仅接收端保存历史；SQLite 存储，含 `folderMode` 字段区分文件/文件夹
- **用户设置**：设备名、端口、限速、并发数、深色模式、下载路径等（SharedPreferences + SQLite）

### 2.6 状态管理 (Riverpod)

主要 Provider（共 9 个）：

| Provider                     | 职责                                        |
| ---------------------------- | ------------------------------------------- |
| `onlineDevicesProvider`      | 在线设备列表 + 上下线事件流                  |
| `connectionStateProvider`    | 连接状态、TransferOffer 接收、配对流程       |
| `activeTransferProvider`     | 当前活跃发送任务 + TransferEngine 生命周期   |
| `transferQueueProvider`      | 等待中的传输任务队列                        |
| `receiveTransferProvider`    | 当前活跃接收任务                            |
| `pendingOfferProvider`       | 待确认的传输请求                            |
| `batteryThermalProvider`     | 电池电量 + 温度状态监控                      |
| `clipboardProvider`          | 剪贴板自动接收订阅                          |
| `settingsProvider` 系列      | 用户配置（名称/端口/限速/深色模式/下载路径） |

另有 `navigationProvider` 管理底部导航 Tab 索引，`platformProvider` 注入平台实现。

### 2.7 平台通道 (PlatformChannel)

- Android 通过 MethodChannel 实现：前台服务、电池/温度、WiFi 锁、content URI 读写、文件打开（FileProvider）、存储权限管理
- Windows 通过 Process.run + FFI 实现
- 存储权限适配：`hasManageStorage` / `requestManageStorage` / `getStoragePermissionGuide` 三个 MethodChannel 方法，针对 HyperOS / ColorOS / OriginOS / MagicOS 提供 ROM 专属设置路径指引

## 三、Engine Isolate 详细设计

### 3.1 设计动机

Dart 单线程事件循环中，文件分片、CRC32 校验、大量磁盘 I/O 等 CPU 密集操作即使使用异步 Future，仍会抢占 UI 线程时间导致卡顿。将传输引擎独立到专用 Isolate，利用多核并保证 UI 线程绝对流畅。

### 3.2 发送端消息通信模型

```
UI Isolate                          TransferEngine Isolate
   │                                    │
   │ ──── start_transfer ────────────►  │  携带文件路径/contentFiles
   │                                    ├─ 扫描文件，获取大小
   │                                    ├─ 判定传输模式
   │ ◄─── file_list_chunk ────────────  │
   │ ◄─── progress ───────────────────  │
   │ ◄─── mode_change ───────────────  │
   │ ◄─── file_complete ─────────────  │
   │ ◄─── error ─────────────────────  │
   │                                    │
   │ ──── pause / resume / cancel ───►  │
   │                                    │
   │ ◄─── transfer_complete ──────────  │
   │                                    ├─ 队列为空 → 销毁 Isolate
```

### 3.3 接收端消息通信模型

```
UI Isolate                          ReceiveEngine Isolate
   │                                    │
   │ ──── accept_receive ──────────►    │
   │                                    ├─ 流式读取 FLP Frame
   │ ◄─── transfer_started ──────────   │
   │ ◄─── progress ──────────────────   │
   │ ◄─── file_complete ────────────    │
   │ ◄─── transfer_complete ────────    │
   │                                    ├─ 销毁 Isolate
```

### 3.4 截断帧防御

`FlpFrame.parse()` 在构建 payload 前校验 `headerLength + declaredLength + checksumLength <= bytes.length`，不满足则抛出 `FlpFrameException` 而非 `RangeError`。发送端使用 `data.length`（实际读取字节数）而非请求大小作为帧头 `dataLength`，防止 partial read 导致的帧长不匹配。

### 3.5 文件扫描与大小

Engine 内部通过 `File.statSync()` 获取实际文件大小（修复早期硬编码 size=0 的问题）。文件夹扫描以 `p.basename(path)` 作为相对路径前缀，确保接收端正确创建目录结构。

## 四、页面结构

| 页面          | 说明                                                |
| ------------- | --------------------------------------------------- |
| **设备与发现** | 在线设备列表 + 发送文件/剪贴板 + 二维码/短码/扫码   |
| **传输**       | 发送/接收传输卡片，显示当前传输文件名、进度、速度     |
| **历史**       | 仅接收端历史；单文件点打开按钮直接打开；文件夹可展开查看文件列表；ROM 存储权限引导 |
| **设置**       | 设备名、端口、限速滑块、并发数、深色模式、下载路径等  |

### 4.1 传输卡片显示

- 始终只显示一个当前传输文件名（非全量文件列表）
- 文件名取 basename，不显示文件夹路径
- 单文件传输 batchName 显示文件名；文件夹显示文件夹名

### 4.2 历史页交互

- 不再使用 ExpansionTile；卡片 trailing 为"打开"按钮
- 单文件：打开按钮 → 系统默认应用打开（Windows: cmd start / Android: FileProvider Intent）
- 文件夹：点击卡片展开文件列表；打开按钮 → 文件管理器定位
- 文件列表中的子文件/子文件夹可点击打开

## 五、通信协议概要（遵循 FLP v1.2）

- **服务发现**：UDP 广播到 `255.255.255.255:45679`
- **传输端口**：TCP 默认 **34568**
- **分块大小**：8MB，滑动窗口 3 chunk（24MB 在途）
- **大文件阈值**：100MB
- **消息格式**：
  - 控制消息：UTF-8 JSON，封装在 FLP Frame 中
  - 数据块：48 字节二进制头部 + 原始数据，零校验和标记
- **强制约束**：UI Isolate 不得直接接触 Socket 及协议解析
